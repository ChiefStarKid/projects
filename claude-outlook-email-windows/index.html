How to let Claude read and send your Outlook email on Windows, without the Graph API

A practical guide · updated 2026-07-15

You want Claude Code to read your inbox, draft replies, chase actions, and keep your calendar in view. What you don't want is to register an Azure app and wire up an OAuth flow just to get there. On Windows there's a much shorter path. This guide connects Claude to the Outlook Desktop App with a single-file Python bridge that reuses the Outlook session you're already signed into. Setup takes about ten minutes, and the only dependency is pywin32.

Why not just use the Microsoft Graph API?

Graph is the right answer for cross-platform, server-side, or headless automation. For a personal AI assistant on your own machine, though, it's a lot of ceremony: an Azure app registration, delegated permissions, an OAuth consent flow, and token refresh handling, all before Claude can read a single email. If Outlook is already open in front of you, that work is redundant. You're signed in; the mailbox is right there.

The bridge takes the other route. It drives the classic Outlook Desktop App through Windows COM automation, the same interface Outlook macros use, so it simply inherits your logged-in session. There are no credentials to configure and no token that can leak.

What you need before you start

  • Windows 10 or 11. COM automation is Windows-only.
  • The classic Outlook Desktop App, signed in and running. This does not work with the new Outlook wrapper, Outlook on the web (OWA), or Mac Outlook.
  • Python 3.9 or later.

How to install the bridge (about 10 minutes)

Clone the repository and install its single dependency:

git clone https://github.com/ChiefStarKid/claude-outlook-bridge.git
cd claude-outlook-bridge
pip install -r requirements.txt

Now smoke-test it against your own inbox. With Outlook open and signed in, run:

python outlook_bridge.py list --pretty

If you see a JSON list of your recent messages, the bridge is talking to Outlook. Every command outputs JSON by default. Add --pretty anywhere for human-readable formatting. Place outlook_bridge.py wherever you'd like to call it from; if you're using Claude Code, ~/.claude/ is a natural home.

How to let Claude Code actually use it

Claude invokes the bridge with a single shell call, python /path/to/outlook_bridge.py [command], so you only need to permit that one command. Add a scoped permission to your project's .claude/settings.json: 

{
  "permissions": {
    "allow": ["Bash(python *outlook_bridge.py*:*)"]
  }
}

This lets Claude run the bridge and nothing else. No other Bash commands get auto-approved. Because it lives in a project's .claude/settings.json, the permission only applies inside that project; delete the line whenever you want to revoke Claude's email access.

What can Claude do with your Outlook once it's wired up?

The bridge exposes the full email lifecycle as simple commands:

CommandWhat it does
foldersList all mail folders.
listList recent messages in a folder.
searchSearch across your mailbox.
read / openRead a specific message or thread.
send / reply / forwardCompose, reply, reply-all, or forward with a note.
move / deleteFile a message into a folder, or remove it.
cal-listQuery your calendar.

In practice that means Claude can read a thread for context, draft a reply in your voice, chase an outstanding action, file the message, and tell you what's next on your calendar. All of it happens inside one conversation, without you switching to Outlook.

Is it safe to let an AI send email?

The bridge is draft-by-default. The send, reply, and forward commands all save to your Drafts folder unless you explicitly add --confirm-send. So Claude can compose freely, but nothing actually leaves your outbox until you've read the draft and approved it. Combined with the scoped permission above, which lets Claude run only the bridge script, you keep a human checkpoint on every outbound message.

Tip: keep the draft-by-default behaviour and review sends yourself. Treat --confirm-send as a deliberate, per-message decision rather than a default you switch on and forget.

When should you use Microsoft Graph instead?

This bridge is deliberately narrow. Reach for the Microsoft Graph API when you need any of the following, because COM automation can't provide them:

  • Headless or server-side automation where no one is signed into Outlook interactively.
  • Cross-platform support (Linux, macOS, containers).
  • Multi-mailbox or non-Outlook accounts, or provider-agnostic email.

For everything else, meaning a local assistant on your own Windows box with Outlook already open, the COM bridge is the shortest path from "I want Claude to handle my email" to it actually working.

Frequently asked

Can Claude read my Outlook email without OAuth or the Microsoft Graph API?

Yes, on Windows. The bridge drives the classic Outlook Desktop App through Windows COM automation, inheriting your already signed-in Outlook session. There's no Azure app registration to create and no OAuth flow to wire up.

Is it safe to let an AI agent send email from my account?

The bridge is draft-by-default: send, reply, and forward save to Drafts unless you pass --confirm-send. Nothing leaves your outbox until you approve it, and Claude Code's scoped permission limits it to running only the bridge script.

Does this work with the new Outlook or Mac Outlook?

No. It needs the classic Outlook Desktop App on Windows 10 or 11, because it uses Windows COM automation. The new Outlook wrapper and Outlook on the web (OWA) aren't supported either. For cross-platform or headless setups, use the Microsoft Graph API instead.

What does it cost?

Nothing. claude-outlook-bridge is open source under the MIT license. Its only dependency is pywin32.
